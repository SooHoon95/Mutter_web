# Inbox/Threads audit

Scope read fully: src/routes/Inbox.tsx, src/routes/People.tsx, src/routes/Create.tsx, src/features/inbox/* (InboxList, SaveToInboxButton, useInbox), src/features/threads/* (ThreadView, CounterpartList, useThreads), src/features/connections/{SendToConnection,useConnections}, src/data/{inbox,threads,connections}.ts, supabase/migrations/0002, 0004, 0006, 0008. Cross-checked every RPC the inbox/thread path calls.

## Findings

[P1] supabase/migrations/0008_exchanges.sql:24-31 (get_my_inbox) + 117-127 (get_thread) — Inbox/threads surface REVOKED and EXPIRED letters as openable. The `inbox` row persists after the sender revokes the delivery link (revoke sets `delivery_links.revoked = true`; it does NOT delete the saved inbox row, and there is no FK from inbox.token → delivery_links). `get_my_inbox` joins only `inbox`+`letters`+`profiles` and `get_thread`'s received-branch joins only `inbox`+`letters` — neither joins `delivery_links` nor filters `revoked`/`expires_at`. Result: the recipient still sees the card with an "열기" link to `/l/:token`, but `get_letter_by_token` (0004:194-201) raises `LINK_REVOKED`/`LINK_EXPIRED`, so the open dead-ends in an error. Same for the ThreadView "받음" open link. → Fix: in get_my_inbox and get_thread's received branch, `join delivery_links dl on dl.token = i.token` and add `where ... and dl.revoked = false and (dl.expires_at is null or dl.expires_at > now())`, OR keep the rows but return a `revoked`/`expired` flag so InboxList/ThreadView can disable the open link and show "만료된 편지". (Audio takedown via `audio_disabled` is handled downstream in the viewer, so that one is fine — only revoke/expiry leak through as broken open links.)

[P2] src/features/inbox/SaveToInboxButton.tsx:6,26 — Saving bypasses react-query cache invalidation, so the 받은 편지함 can show stale data. The button imports `saveToInbox` directly from `@/data/inbox` and calls it (line 26), instead of using the `save` mutation from `useInbox` (useInbox.ts:27-33) which invalidates `['inbox']` on success. After a recipient saves a letter from the Viewer and navigates to /inbox, the `['inbox']` query may serve a cached list missing the just-saved letter until refetch/staleTime expires. The local `saved` state masks this on the Viewer page only. → Fix: route the save through `useInbox().save` (or call `queryClient.invalidateQueries({ queryKey: ['inbox'] })` after `saveToInbox`). Note: `useInbox`'s exported `save`/`isSaving` are currently dead code because of this.

[P2] src/features/threads/useThreads.ts:8,39 & src/features/connections/useConnections.ts:16,69 — Thread cache invalidation after a direct send is a partial-key match that works only by react-query default behavior, and is fragile. `useThread` uses key `['thread', counterpartId]` while the send mutation invalidates the base key `['thread']`. This relies on react-query's default partial (prefix) matching — correct today, but if anyone adds `exact: true` or changes the key shape it silently breaks the "send → thread refreshes" flow. Low risk, flag only. Counterparts invalidation (`['counterparts']`) is exact and fine.

## Verified correct (no defect)

- 답장 button null-guard (InboxList.tsx:49) — `{item.senderId !== null && <Link to={`/create?to=${item.senderId}`}>}`. Null senderId (no-account/unknown sender) correctly hides the 답장 link. No `null`/`undefined` ever reaches the URL. ThreadView.tsx:78 uses `counterpartId` which is non-null by construction (only rendered when People's `selected !== null`, and counterpart_id is a real user uuid from get_counterparts).

- Reply to a NON-connection does not break. Create.tsx:30-39 reads `?to=`, checks membership in `connections` (matching `c.userId === replyTo`; Connection.userId confirmed in data/connections.ts:57,81), and auto-switches `sendMode` to 'link' when the target isn't a connection (useEffect 35-39), showing the 전달 링크 path. SendToConnection.tsx:28-33 independently clears a stale `preselectId` that isn't in `connections`, so even if that tab is opened the 보내기 button can't fire a NOT_CONNECTED send (button disabled when selectedId===null). Double-guarded. `/preview/:id` (ThreadView sent target) and `/create` both exist in router.tsx:55,60.

- Thread direction labels (보냄/받음) are correct. get_thread (0008:118-126): branch 1 `l.owner_id = auth.uid()` → 'sent' → ThreadView renders 보냄 (I own it = I sent). Branch 2 `i.user_id = auth.uid()` → 'received' → 받음 (I saved it = I received). openTarget (ThreadView.tsx:25-30): sent→/preview/:letterId (token intentionally null), received→/l/:token with explicit null-token guard (line 29, hides 열기 if token missing). Matches the SQL contract comment.

- Thread ordering correct. `order by at` at the tail of the UNION ALL (0008:127) binds to the whole combined result in Postgres, so the merged sent+received thread is chronologically sorted, not per-branch.

- save_to_inbox idempotency (0006:83-86) — `insert ... on conflict (user_id, letter_id) do update set token=excluded.token, saved_at=now()`. Re-saving the same letter updates token/timestamp instead of duplicating. Validates token exists and not revoked before insert (0006:79-81). No dup rows possible (unique constraint 0006:61).

- RLS / cross-user leakage — all four RPCs (get_my_inbox, get_my_sent_with_recipients, get_counterparts, get_thread) are `security definer set search_path=public`, each raises `AUTH_REQUIRED` when `auth.uid() is null`, and every query branch is anchored to `auth.uid()`. get_thread takes an arbitrary `p_counterpart` but BOTH union branches still require my own uid as owner/saver, so passing someone else's id can only return rows that already involve me — no way to read a thread between two other users. profiles is reached only via security-definer join (RLS on profiles blocks direct reads). No leakage found.

- null nickname / null sender handling — every render site coalesces: InboxList.tsx:44 `senderNickname ?? '알 수 없음'`, CounterpartList.tsx:50 `nickname ?? '알 수 없음'`, plus left joins on profiles in SQL mean null is expected, not a crash. formatDate receives savedAt/lastAt/at which are NOT NULL columns (saved_at default now(), created_at not null), so no `Invalid Date` from those paths. (get_my_sent_with_recipients can return null saved_at, but that RPC feeds the Sent screen, out of inbox/thread scope.)
