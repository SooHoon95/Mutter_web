P0 src/features/viewer/fallbackSourceFactory.ts:84-86 + src/features/music/SyncEngine.ts:99-119 — **iOS silent-letter on the PRIMARY (SoundCloud) path.** The whole iOS unlock contract is: call the real `play()` *synchronously inside the click tick* so iOS permits later auto-play. SyncEngine.unlockAll does this by calling `src.unlock()` in-gesture (line 107) BEFORE awaiting loadPromise. But every cue is wrapped in `FallbackTrackSource`, whose `unlock()` is `return this.active?.unlock() ?? Promise.resolve()` — and `this.active` is set only AFTER `load()` resolves (lines 65-82). For a SoundCloud cue, `load()` cannot resolve in the click tick: it must inject the SC iframe, load the Widget API script, and wait for the READY event (up to READY_TIMEOUT_MS = 8000ms). So at gesture time `this.active === null` and `unlock()` is a **no-op** — no `play()` runs inside the user gesture. The actual play happens later in `activate()` (SyncEngine.ts:166-172) *outside* the gesture tick, which iOS Safari blocks → the SoundCloud track (the v1 primary pillar) plays no sound on iOS even though SC succeeded. CC0 fallback shares the same gap. → Fix: the FallbackTrackSource must own an iframe/audio element that exists at construction time so its `unlock()` can synchronously fire a real `play()` in-gesture; OR have unlockAll re-issue `src.unlock()` again right before/inside `activate()` is insufficient (still outside the tick). Concretely: create the primary source eagerly in `attach()` (already done) AND make `FallbackTrackSource.unlock()` prime the underlying media synchronously — e.g. start `load()` at construction and, when `active` is still null at unlock time, queue the unlock so it runs the instant `active` is assigned, and rely on a pre-created `<audio>`/iframe whose `.play()` was already invoked in-gesture. This needs a real device test; the current code path provably never calls play() in-gesture for SC.

P1 src/features/viewer/AudioUnlockGate.tsx:35-43 — **Gate can get stuck on "여는 중…" forever (button permanently disabled).** `handleOpen` sets `opening=true` and never resets it; `onUnlock()` rejection/hang is swallowed (`void onUnlock()`). If `unlock()` → `unlockAll()` throws (it shouldn't today, but `engineRef.current?.unlockAll()` could reject if a future source throws synchronously) the button stays `disabled` showing "여는 중…" with no retry. More importantly, even on the happy path the gate is removed by the parent (`!unlocked`) so this is usually masked — but the parent removes the gate via the `unlocked` state set in `handleUnlock`, NOT via gate-local state. If `setUnlocked(true)` and the unmount happen but `onUnlock` rejects, there is no user-visible failure path. → Fix: wrap `onUnlock()` in `.catch()` and reset `opening` on failure so the user can retry; do not rely solely on parent unmount.

P1 src/features/viewer/LetterView.tsx:83,107,137 vs SyncEngine.attach:73-75 — **"무음 편지 0" can break: a letter whose only cue is invalid still shows the player as if music exists, but no fallback is triggered.** `hasMusic = !audioDisabled && cues.some(c => c != null)` and the top player renders when `hasMusic && unlocked`. SyncEngine.attach uses `cues.find(c => c != null)` to pick the source and the FallbackTrackSource guarantees CC0 only when a cue object EXISTS. That is consistent. BUT `normalizeCue` (useLetterViewer.ts:80-90) drops any cue with a bad `sourceType` or non-string `ref` to `undefined`. So a letter authored with a single malformed cue yields `cues = [undefined,...]` → `attach` finds no cue → `source` stays null → unlockAll returns false → genuinely silent letter, while the "무음 편지 0" invariant intends a CC0 floor. The fallback only fires for cues that survive normalization and then fail at load time, never for cues that fail normalization. → Fix: decide the product rule — if a letter is supposed to always have music, the fallback floor should be applied at the viewer level (e.g. when `cues` has no valid entry but the letter is not `audioDisabled`, inject a default CC0 cue) rather than only inside FallbackTrackSource.

P1 src/features/music/SyncEngine.ts:138-147 (resume) + useScrollSync.ts:56-66 (togglePlay) — **resume() after a failed unlock plays nothing but the UI flips to "재생 중".** If `unlockAll()` returned false (no source, or load failed in a way that left `started=false`), `userPaused` is false and `unlocked` is true (set at unlockAll:100 unconditionally). A later `togglePlay` reads `engine.paused` (false) → calls `engine.pause()` → sets isPlaying false; next toggle calls `resume()` which early-returns only if `!this.unlocked || !this.source`. With a null source it no-ops, but `useScrollSync.togglePlay` still `setIsPlaying(true)` on the resume branch → false "재생 중" UI with silence. The top player only renders when `hasMusic`, which requires a non-null cue, so this is reachable when a cue exists but its source load path ends with `started=false` (e.g. pause raced the unlock). → Fix: gate the player toggle on `started`/actual playback, or have `resume()`/`togglePlay` report back whether playback actually resumed and reflect that in `isPlaying`.

P2 src/features/viewer/LetterView.tsx:52-55 — **Global SoundCloud container is a shared mutable singleton; concurrent/overlapping viewer mounts clobber it.** `setSoundCloudContainer(scHostRef.current)` writes a module-global in SoundCloudSource. With route transitions or React 18 transitions, an unmounting LetterView's cleanup `setSoundCloudContainer(null)` can run AFTER a newly mounted LetterView set it, nulling the active container so the next SC iframe falls back to `document.body` (the exact iOS-PWA无音 case the comment warns about). Single-viewer usage is fine; this is a latent ordering hazard. → Fix: pass the container explicitly via SoundCloudSourceOptions/factory instead of a module global, or ref-count the global.

P2 src/features/viewer/LetterView.tsx:45 — **`sourceFactoryRef = useRef(createFallbackSourceFactory())` calls the factory builder on every render** (the argument is evaluated each render even though only the first result is kept). `createFallbackSourceFactory()` reads the catalog (`getCatalog`) lazily only when a cue is built, so the per-render cost is just object allocation — harmless functionally but wasteful. → Fix: `useRef<...>(); if (!ref.current) ref.current = createFallbackSourceFactory();` or `useMemo`. (Note: behavior is correct because only `ref.current` from the first render is used; flagging only as minor.)

P2 src/features/music/SyncEngine.ts:113-117 — **`this.unlocked` is set true at unlockAll:100 and never reset on the false/no-source path, and `activate` is guarded by `this.unlocked && !this.userPaused`.** Combined with the resume issue above, the engine considers itself "unlocked" even when nothing ever played. Low impact today (single unlock per mount) but couples with P1 resume bug. → Fix: only set `unlocked` semantics from actual playback start, or track `started` distinctly everywhere the UI reads play state (partially done — `started` exists but UI uses `unlocked`-derived `isPlaying`).

## Verified clean (no defect found)

- **Read-receipt fires exactly once per genuine open (not 0×, not 2×).** LetterView.tsx:61-69 guards with `openRecordedRef` and the effect deps `[token, unlocked, audioDisabled]`. Normal letter: nothing on mount (gate covers body), fires once when `unlocked` flips true. `audioDisabled` letter: fires once on mount (no gate). `token === undefined` (Preview never renders LetterView; only Viewer passes token): never fires. StrictMode double-mount (main.tsx:24) is dev-only and the ref+server-side `record_letter_open` no-op-on-invalid keep it idempotent in dev too. Confirmed correct.

- **Preview.tsx does NOT fire a read-receipt.** Preview uses `getLetter(id)` + TemplateThemed/Paginated only (routes/Preview.tsx:60-66). It never imports recordLetterOpen, never renders LetterView, never mounts audio. No false receipt. Correct per design (owner static preview).

- **notYet does not leak password-protection status.** Viewer.tsx:33-60 renders the sealed screen purely from `status==='notYet'` + optional `revealAt`; identical output whether or not the link has a password. links.ts:291-296 converts `NOT_YET_REVEALED:<ISO>` to LinkNotYetError before any password normalization, and useLetterViewer.ts:163-168 short-circuits to `notYet` before the WRONG_PASSWORD branch. No leak.

- **LinkNotYetError parsing** links.ts:294-295 — regex `/NOT_YET_REVEALED:(.+)$/` with `.trim()` correctly extracts the ISO timestamp; `revealAt` is surfaced and rendered via `toLocaleString('ko-KR', …)`. `revealAt` null is handled (Viewer.tsx:34-43 guards `revealText`). Correct.

- **useLetterViewer unmount race** is handled via `aliveRef` (useLetterViewer.ts:143,156,161,179) — no setState after unmount. The `attempt` useCallback depends only on `[token]`, and the load effect deps `[attempt]` re-runs cleanly when token changes (resets `aliveRef.current = true` at effect start). Correct.

- **SoundCloudSource listener/iframe leak** — destroy() (SoundCloudSource.ts:210-225) unbinds all four events (READY/ERROR/PLAY_PROGRESS/FINISH), clears cb sets, removes the iframe, nulls refs. HostedAudioSource.destroy() (HostedAudioSource.ts:87-97) removes all tracked listeners, pauses, clears src, reloads. useScrollSync cleanup (useScrollSync.ts:43-46) calls engine.destroy() which calls source.destroy(). No leak found. (The only iframe-orphan risk is the document.body fallback from the P2 container issue, not a leak per se.)

- **Credits attribution (license-compliance)** Credits.tsx:62-85 collects de-duped entries; CC-BY → `attributionRequired`; hosted cue not in catalog (fallback CC0) is skipped (no attribution obligation for CC0). SC cue shows neutral source URL. Correct. (Note: Credits renders from the AUTHORED `cues`, not the actually-played source. If a SC cue falls back to CC0 at runtime, Credits still shows the SC URL and NOT the CC0 track. This is arguably the correct compliance behavior since CC0 needs no attribution, but it means the credit can name a track that never played — informational, not a defect.)

- **useScrollSync engine re-create race** — effect deps `[cues, sourceFactory]` (useScrollSync.ts:47). `cues` is memoized in LetterView (`effectiveCues`, LetterView.tsx:75) and `sourceFactory` is the stable ref. Flipping `unlocked` does NOT re-run the engine effect, so the engine is not destroyed/recreated on unlock (the comment's stated hazard is correctly avoided). Verified.

- **togglePlay stale-state** reads `engine.paused` live from the instance (useScrollSync.ts:58-65), not a captured React value — no stale closure.
