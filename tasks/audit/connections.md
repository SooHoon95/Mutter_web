## Findings

[P1] supabase/migrations/0013_connections_grants.sql:110-119 (accept_connect_invite) — Race breaks the 독점 1:1 invariant. Exclusivity is enforced only by two `EXISTS` reads ("am I connected? / is inviter connected?") followed by an `INSERT … on conflict (user_a,user_b) do nothing`. The unique constraint is on the *pair* `(user_a,user_b)` (0009:28) — there is NO per-user uniqueness/exclusion constraint and no advisory lock / SERIALIZABLE / row lock anywhere (grep confirms none). Under Postgres default READ COMMITTED, if user B holds two pending invites (from A and from C) and accepts both near-simultaneously, both transactions run the EXISTS checks before either commits, both see "no connection", both INSERT distinct pairs → B is now connected to BOTH A and C. Same hazard if A and B accept each other's invites concurrently against a third party. Result: a user with 2+ live connections, violating the core promise. → Fix: enforce at the DB level, e.g. add an exclusion/partial-unique guard so each user_id can appear in at most one connections row (a normalized "membership" unique, or `create unique index on connections(user_a)` + `(user_b)` is insufficient since a user can be user_a in one row and user_b in another — instead add a per-user constraint via a membership table or check, or take `pg_advisory_xact_lock(hashtext(least||greatest))` AND lock both users' rows). Simplest robust fix: `pg_advisory_xact_lock` on each of the two user UUIDs at the top of accept_connect_invite, then re-check, so concurrent accepts serialize.

[P2] supabase/migrations/0009_connections.sql:50 vs 0013:48 (create_connect_invite) — No expiry/dedup/cap on invite creation: every call inserts a fresh row. `token` is PK so collisions are impossible, but a user can accumulate unbounded outstanding invites, and there is no "revoke old invite when creating a new one." Not a security hole (each token is 128-bit unguessable and 7-day-expiring in 0013), but stale invite links remain valid for 7 days even after the inviter forms a different connection — opening them just yields ALREADY_CONNECTED_SELF/INVITE state messaging, which is handled gracefully in Connect.tsx. Low priority. → Optional: expire/clear the inviter's outstanding invites on connect, or cap rows per inviter.

[P2] supabase/migrations/0013_connections_grants.sql:121 (accept_connect_invite) — `update connection_invites set accepted_by/accepted_at` runs even when the `insert … on conflict do nothing` was a no-op (already-connected pair re-accept). Harmless (the EXISTS guards above already reject the re-connect with ALREADY_CONNECTED_SELF before reaching the insert for the *other-connection* case), but for the *same-pair* re-accept path the guard at :110 fires first, so this update line is effectively unreachable for an existing pair. No defect; noted for completeness — no fix needed.

## Verified CLEAN (end-to-end loop + auth/grants)

End-to-end loop traced and sound:
- **Create invite**: InvitePanel → useConnections.createInvite → data.createInvite → generateToken (≥128bit, 22+ chars, url-safe; src/lib/token.ts) → RPC create_connect_invite. RPC validates auth + token shape (`length>=22`, `^[A-Za-z0-9_\-]+$`) and writes inviter_id=auth.uid() with 7-day expires_at (0013:48). buildInviteUrl → `/connect/:token`. ✔
- **Open invite**: Connect.tsx (router-guarded, withProtectedCreatorShell → RequireAuth; router.tsx:61) → getInvite → get_connect_invite. RPC requires auth, INVITE_NOT_FOUND on missing/expired (0013:75-79), returns 6 cols: is_self, already_connected, viewer_has_connection, inviter_has_connection. Connect.tsx renders all 5 states (self / already / viewer-blocked / inviter-blocked / acceptable) correctly. ✔
- **Self-invite guard**: get_connect_invite returns is_self (`v_inviter = auth.uid()`); accept_connect_invite hard-rejects with CANNOT_CONNECT_SELF (0013:107). ConnectInviteRow maps `viewer/inviter_has_connection === true` so a legacy 4-col deploy fails safe to false but exclusivity is still enforced server-side in accept. ✔
- **Accept**: useConnections.acceptInvite → accept_connect_invite. Normalizes pair via least/greatest, `check(user_a<user_b)` + `unique(user_a,user_b)` prevent dup pair & self-pair. Exclusivity checked (modulo the P1 race above). Error codes ALREADY_CONNECTED_SELF/OTHER/CANNOT_CONNECT_SELF/INVITE_NOT_FOUND normalized to Korean in data layer; covered by connections.test.ts. ✔
- **Connection list**: get_my_connections (auth-gated) returns the other party via `case when user_a=uid then user_b else user_a`, joins profiles for nickname. ConnectionList/ConnectionPicker/InvitePanel all read the shared ['connections'] query; isLoading gating in InvitePanel prevents the "invite UI flash" for already-connected users. ✔
- **Send-to-connection**: SendToConnection → ConnectionPicker (radio over real connections) → sendToConnection → send_to_connection RPC. RPC enforces: auth; CANNOT_SEND_SELF (0013:136); FORBIDDEN unless `letters.owner_id=auth.uid()` (can't send someone else's letter); NOT_CONNECTED unless a real connections row for the normalized pair exists (delivers to the RIGHT person — recipient must be the connected party); TOKEN_INVALID on bad token. Inserts delivery_links (owner=sender, unique token) + recipient inbox row with `on conflict (user_id,letter_id)` (constraint exists, 0006:61). preselectId is validated against the live connections list in SendToConnection.tsx:28-33 AND Create.tsx:32-39 falls back to link mode if reply target isn't connected — preselect cannot reference a non-connection. Sender sees the exchange via get_thread/get_counterparts (0008, sender side = letters JOIN inbox where owner=uid). ✔
- **Disconnect**: disconnect_connection deletes the user's connection row(s) only; letters/inbox/delivery_links untouched (0010:78) → threads preserved as designed. UI confirms before disconnect. ✔

RPC auth/grants cross-check (0009/0010/0013): every connection RPC is `security definer set search_path=public`, checks `auth.uid() is null → AUTH_REQUIRED`, and has explicit `revoke all … from public` + `grant execute … to authenticated`. 0013 re-asserts grants for all six RPCs (closing the 0010 gap where the redefined accept_connect_invite had no explicit grant block). Tables `connection_invites` and `connections` have RLS ENABLED + FORCE; connection_invites policy = inviter-only (`auth.uid()=inviter_id`), connections policy = members-read-only, writes only via security-definer RPC. anon has no execute and no table access. No anon/cross-party read or mutate path found. ✔
